## Часть 1. Подготовка и повторение теории (FastAPI + JWT)

### Структура backend-приложения и слои
В проектах на FastAPI обычно удобно разделять код по смысловым слоям:
- `routers` — принимают HTTP-запросы, описывают эндпоинты, статус-коды и dependency.
- `schemas` (Pydantic models) — описывают входные/выходные данные, обеспечивают валидацию и сериализацию.
- `models` — доменные модели данных (например, пользователь в БД).
- `services / utils` — бизнес-логика: правила аутентификации, операции над данными, работа с JWT и паролями.
- `repository` — слой доступа к данным (БД/хранилище). Он изолирует источник данных от остального приложения.

Такое разделение упрощает тестирование: роуты не содержат “тяжёлую” логику, а services не завязаны на HTTP.

### Dependency injection в FastAPI
FastAPI использует `Depends(...)` для DI:
- dependency описывает функцию, которая будет выполнена перед роутом,
- результаты dependency прокидываются в параметр роутера.

В JWT-проектах dependency часто используется для проверки токена и извлечения пользователя.

### FastAPI: роутеры, Depends, response models
- `APIRouter` группирует эндпоинты и позволяет подключать их в `main.py` через `include_router`.
- `Depends` подключает dependency (например, функцию извлечения пользователя из JWT).
- `response_model` заставляет FastAPI сериализовать ответ по Pydantic-модели и обеспечивает предсказуемую структуру ответа.
- `status_code` задаёт HTTP-код успеха/ошибки для роутера.

### Pydantic: валидация и сериализация
Pydantic гарантирует:
- валидацию входных данных (например, `LoginRequest` не примет неверный тип),
- сериализацию ответа в нужный формат через `response_model`.

### JWT авторизация: структура и жизненный цикл
JWT — компактный JSON Web Token, обычно используется как access token:
- `header` — тип токена и алгоритм,
- `payload` — полезные данные (в этом проекте: `sub` + `exp` + `iat`),
- `signature` — подпись, подтверждающая целостность и подлинность токена.

Access token имеет срок жизни (`exp`). При истечении или ошибке подписи защищённые роуты возвращают `401`.

Refresh token в этом проекте не используется (в задании было указано как теория, но в коде — только access token).

### Авторизация и Bearer Authentication
Клиент отправляет токен так:
- `Authorization: Bearer <access_token>`

FastAPI извлекает токен из заголовка (через `HTTPBearer`), валидирует и после этого позволяет выполнить защищённый endpoint.
